# 009: Sprint 3 指标版本化 + 策略 DSL + 合约规格抽象

日期: 2026-03-05
状态: accepted

## 背景

Sprint 3 需要三个能力同时到位：

1. 指标注册表支持版本号，便于后续同名指标演进与回放一致性。
2. 用 YAML DSL 描述策略，避免硬编码参数组合，且必须覆盖 V4 的 framework/risk_mode 组合。
3. 提供合约规格抽象，尤其是 XAUUSD 的乘数、最小跳动和交易时段，不再散落在业务代码中。

## 选项

- 选项 A：最小可用抽象（本次采用）
  - 指标注册表扩展为 name + version 二级索引，默认获取 latest。
  - 策略 DSL 采用 `strategy.name/framework/risk_mode/params/weight` 结构，提供解析与实例化函数。
  - 合约规格通过独立模块维护默认值，并支持配置覆盖。
  - 优点：改动面可控、向后兼容、能立即支撑实验与配置化。
  - 缺点：暂未把合约规格全面接入引擎执行路径。

- 选项 B：一次性全链路重构
  - 同时改造 config、engine、report、audit 使用统一合约规格对象。
  - 优点：架构完整。
  - 缺点：改动面大、回归风险高，不适合当前 Sprint 节奏。

## 决定

采用选项 A。先完成稳定可用的基础抽象与 API，对外保持兼容，后续迭代再推进全链路接入。

## 后果

- 正面：指标演进与策略配置化能力落地，V4 组合可由 YAML 描述，合约规格有统一入口。
- 负面：引擎侧仍以 `contract_multiplier` 为主，需在后续迭代中接入 `contracts/spec.py` 以实现端到端配置驱动。
